<a id="q1-generated-api-evolution"></a>
## Q1: How Can a Macro Update Break Clients?

### Short Answer

Even unchanged invocation syntax can generate renamed members, new overload ambiguity,
different access or isolation, changed conformances, or altered Codable/persistence
behavior. Version and test invocation and expansion separately, compile representative
consumers, and provide deprecation or parallel versions for migration.

### Expanded Answer

Source, binary, serialized, and operational compatibility are distinct. Persisted
formats generated by a macro need explicit schema policy rather than implicit expansion coupling.

### Trade-offs

- Central updates fix many call sites at once.
- The same leverage amplifies regressions.
- Parallel versions ease rollout but extend maintenance.

### Example

A macro changes generated coding keys and old records stop decoding. The corrected
release restores stable keys and introduces a versioned opt-in expansion.

## Q2: How Should an Organization Govern Macro Adoption?

### Short Answer

Define approved use cases, package and toolchain compatibility, dependency/security
review, generated-API review, build-time budgets, diagnostic standards, owners, rollout,
rollback, and retirement. Track incremental and clean build cost, plugin failures,
cache behavior, diagnostics, and adoption. Require a nonmacro alternative analysis.

### Expanded Answer

Macros run in the compiler workflow and affect every consumer's developer experience.
Central packages need funded support and compatible releases before toolchain upgrades.

### Trade-offs

- Shared macros enforce consistency and remove repetition.
- Central compiler dependencies create broad failure domains.
- Governance slows experimentation while protecting fleet productivity.

### Example

A macro package blocks a Swift upgrade across dozens of modules. A compatibility matrix,
canary consumers, and rollback artifacts make future upgrades incremental.
